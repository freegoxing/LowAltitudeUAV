# 面向低空复杂环境的态势感知与全链路协同控制流程

## 1. 核心业务流程概述

你总结的流程非常准确，而且将最初的“物理路径规划”进一步拓展到了“通信资源流向规划”，这在6G语义通信架构中极具创新性。整个系统的工作流可以概括为一条**从“边缘语义提取”到“中枢认知决策”，再到“底层策略执行”的闭环链路**。

完整的流水线如下：
**通信节点反馈 (语义Token)** $\rightarrow$ **动态知识图谱构建 (KG)** $\rightarrow$ **图神经元感知 (HGT)** $\rightarrow$ **双LLM智能体 (态势与决策)** $\rightarrow$ **强化学习执行 (RL 资源与路径规划)**。

---

## 2. 态势感知与全链路完整工作流

### 步骤一：边缘侧通信节点的语义Token化反馈（避免全量数据暴增）
* **动作：** 通信节点（无人机、地面终端等）不再回传海量的全量信道CSI，而是通过部署在边缘侧的轻量级模型对核心通信指标进行**特征提取**。
* **数据流向：** 将提取的关键信息封装为离散的 **语义Token**（如：`[节点ID: UAV-3, 节点类型: UAV-M, 信噪比状态(SNR): 15dB]`）。根据实际代码落地情况，这里**仅考虑信噪比 (SNR) 状态**作为核心感知特征。
* **意义：** 极大地降低了上行通信带宽的压力，并聚焦于通信质量本身。

### 步骤二：异构动态语义知识图谱（KG）建模（态势的具象化）
* **动作：** 指挥中心或算力节点接收到各个节点的语义Token后，将其映射到一张包含实体（Entity）和关系（Edge）的**异构图谱**上。
* **实体节点（Nodes）：** 参与通信的各类设备（如任务无人机、中继无人机、地面终端、基站等）。
* **关系边（Edges）：** 节点间的通信链路质量。由于实际场景主要聚焦于通信态势，关系边的权重与属性**完全由节点间的 SNR 状态决定**。
* **意义：** 此时的图谱就是当前低空环境通信质量的**“结构化态势”**。它随着各节点 SNR 的变化而动态演化。

### 步骤三：HGT（异构图Transformer）语义感知压缩
* **动作：** 原始的图谱虽然去除了冗余数据，但对于下游网络和LLM来说维度仍然较高。此时引入 **HGT (Heterogeneous Graph Transformer)**。
* **处理机制：** HGT 利用**注意力机制（Attention）**聚合节点信息。例如，当评估某无人机的通信态势时，HGT 会自动基于 **SNR 状态**，对 SNR 极度恶化或作为关键中继瓶颈的节点加大注意力权重，忽略 SNR 良好且对当前链路无关的节点。
* **输出：** HGT 将仅包含 SNR 状态的复杂通信图谱，压缩成一组高密度的**图特征向量（Graph Embedding）**和**特征Token**，作为态势感知的基础输入。

### 步骤四：双LLM智能体协同认知（态势分级与任务翻译）
这一步是整个系统的“大脑”。
* **Agent 1（态势评估与任务分级）：**
  * **输入：** HGT 产生的图特征Token + 人类指挥官的自然语言宏观指令。
  * **动作：** 基于历史经验和内置逻辑规则，仅根据 HGT 提取的**全网 SNR 分布特征**，对当前通信态势进行**打分（如通信中断风险、链路紧急度）**。
  * **输出：** 输出通信态势级别（如 Level 1 严重断连风险、Level 2 常态波动等），并生成一份结构化的态势研判报告。
* **Agent 2（对话决策与参数翻译）：**
  * **输入：** Agent 1 的态势报告 + HGT 状态 + 用户的对话指令。
  * **动作：** 将宏观的、人类可读的决策意图，翻译为机器可读的数学参数。
  * **输出：** 生成下发给底层控制器的**强化学习奖励函数权重矩阵 $\mathbf{W}$**（例如：在通信受阻的 Level 1 态势下，调高“通信质量维持”的权重，降低“能耗优先”的权重）。

### 步骤五：强化学习（RL）多维资源与路径规划执行
* **输入：** HGT 提取的紧凑图状态向量 $\mathbf{S}_{graph}$（作为 RL 的 State） + Agent 2 下发的奖励权重矩阵 $\mathbf{W}$（定义 RL 的 Reward）。
* **动作：** RL 智能体（如 PPO 或 SAC 算法）在设定的状态空间中探索，根据最大化期望奖励来生成具体的控制指令。
* **输出（Action）：**
  1. **通信资源流向规划：** 调整发射功率、动态分配带宽，或选取特定的通信路由，确保高优先级任务的链路 SNR 达标。
  2. **中继与物理辅助（可选）：** 如果控制空间包含物理动作，也可指引中继无人机向 SNR 恶化的区域移动，以物理位置的改变来优化通信网络整体的 SNR 态势。

## 3. 如何在代码和架构中落地“态势感知”？

为了将上述流程在工程上实现，你可以采取以下模块化的设计：

1. **图谱维护模块 (Graph Manager)：**
   写一个后台服务（比如用 NetworkX 或 PyTorch Geometric），专门负责接收各个通信节点的 UDP/TCP Token 报文，并实时更新图的节点属性和边权重。
2. **HGT编码模块 (Perception Layer)：**
   使用 PyTorch 构建一个 HGT 层，定期（比如每 1 秒）对当前图谱进行一次 Forward 传播，拿到 `graph_embedding`。
3. **态势评估队列 (LLM Async Queue)：**
   LLM 推理速度较慢，不能放在实时的物理控制循环中。应该将 `graph_embedding` 转化为文本 Prompt（例如 JSON 描述），每隔一段时间（如 10 秒）或者当图谱发生剧烈变化（如关键节点掉线）时，异步触发 Agent 1 进行态势重新评级。
4. **控制循环层 (RL Loop)：**
   在底层保持高频（如 50Hz）的 RL 决策循环，它使用最新的 `graph_embedding` 和 LLM 最近一次下发的 `权重 W`，持续输出具体的通信分配和飞行控制动作。
